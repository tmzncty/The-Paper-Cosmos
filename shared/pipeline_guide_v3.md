## 二、写作流水线（标准流程）

### 2.1 每批5章的三步法

```
批次N: 赫默写5章 → 符玄逻辑审计 + 真理事实核查 → 赫默返修 → commit+push
```

**关键规则**：
1. **每批5章**。不要一次写太多——审稿压力会堆积
2. **双审并行**。符玄和真理可以同时审不同维度，不互相阻塞
3. **审稿Session一次性用完即弃**，每批新建
4. **有实质性产出就 git commit + push**

### 2.2 Session分发模板

分发写作任务给赫默时，一次性给足上下文：

```markdown
## 任务：《纸上宇宙》chXX-XX 初稿写作
### 项目路径 + 大纲参考 + 章节规划（表格）
### 知识点覆盖（每章★核心+●重点+○扩展）
### 写作规则（硬性要求）
### 参考前文（衔接文件路径）
### 角色一致性
### 文件命名 + 产出
```

分发审稿任务时，一次性给足所有需要的信息：

```markdown
## 任务：《纸上宇宙》chXX-XX 逻辑审计/事实核查
### 项目路径 + 审计范围 + 大纲参考
### 审计维度 + 格式参考 + 产出
```

## 三、写作规则（硬性要求）

| 规则 | 阈值 | 说明 |
|------|------|------|
| 每章字数 | ≥5,000字 | 正文+知识清单合计 |
| 破折号（——） | ≤15个/章 | bash: `grep -o "——" file \| wc -l` |
| "够了" | ≤20次/章 | 递进式句法的点睛用法，不是每段都要 |
| "是最" | ≤15次/章 | "X是最Y的"句式——容易变成定义链 |
| "就是" | ≤8次/章 | 可替换为"正是"/"便是"/"不过是" |
| "是...的"句式链锁 | 不超过2句连续 | 番外一的惨痛教训：11.3% → 降至5%以下 |
| 每章末尾知识清单 | ★核心+●重点+○扩展 | 考研知识点融入方式：NPC对话40% / 环境25% / 情节20% / 独白15% |

### 递进式句法风格

核心骨架：`"A不是B。A是C。C意味着D。"`
- 保持这个递进骨架，但要在递进中穿插描写、对话、动作
- **不能**让递进变成机械的词典式定义链
- 参考番外五的写法——句式多样且流畅

### 防重复文字机制

2026-06-22发现全书存在系统性重复文字问题后，建立了以下机制：

1. **写作阶段**就遵守阈值（不要写到审稿才发现）
2. **审稿阶段**必须做重复文字密度审计（见第四节）
3. **返修阶段**用脚本+EDIT结合的方式批量处理

## 四、审稿规范（详细）

### 4.1 符玄逻辑审计维度

1. **大纲遵循度** — 标题、视角分配、知识点覆盖是否与大纲吻合
2. **逻辑一致性** — 跨章引用链、空间轨迹、角色行为逻辑
3. **知识准确性** — 知识清单中的事实声称（与真理核查互补）
4. **叙事弧完整性** — 5章作为一个叙事单元是否闭合
5. **角色一致性** — 绯墨和苏墨的行为是否一致
6. **跨章衔接** — 与前后批次的过渡
7. **重复文字密度审计** — 见下表

### 4.2 真理事实核查维度

1. 逐章提取知识清单中的所有**可核查事实声称**
2. 对每条声称进行验证：
   - ✅ [已验证]
   - ⚠️ [需修正] — 措辞不精确、年份有争议、归类偏差等
   - 🔴 [与源矛盾] — 事实硬伤
3. 跨章一致性检查
4. 直接引语准确性
5. **重复文字密度审计** — 见下表

### 4.3 重复文字密度审计（v2.0新增）

**所有审稿Session必须在报告末尾增加此章节**：

| 检查项 | 阈值 | 统计命令 |
|--------|------|----------|
| `够了`（含"就够了"） | ≤3次/千字CJK | `grep -o "够了" file \| wc -l` |
| `是最` | ≤3次/千字CJK | `grep -o "是最" file \| wc -l` |
| `不需要` | ≤2次/千字CJK | `grep -o "不需要" file \| wc -l` |
| `不是…是`定义式 | ≤5次/千字CJK | 脚本统计 |
| `就是` | ≤1次/千字CJK | `grep -o "就是" file \| wc -l` |

**报告格式**：
```markdown
## X、重复文字密度审计
| 模式 | 次数 | CJK字数 | 密度(/千字) | 阈值 | 判定 |
```

### 4.4 判定标准

| 判定 | 含义 | 后续动作 |
|------|------|---------|
| ✅ 通过 | 无问题或仅有🟢可忽略问题 | 直接入库 |
| ⚠️ 有条件通过 | 有🟡需修正问题 | 返修后不需要重新审稿 |
| 🔴 不通过 | 有🔴严重问题 | 返修后必须重新审稿 |

### 4.5 常见错误类型

| 类型 | 示例 | 严重度 |
|------|------|--------|
| **事实硬伤** | 老舍"留在沦陷区"（实际去了重庆）、《骆驼祥子》年份1930（实际1936） | 🔴 P1 |
| **元叙事泄露** | 角色说"第九十二章那次"——角色不应该知道章节号 | 🔴 P1 |
| **系动词误删** | 修复"是最"时把"是"一起删了 → "方向极其好的东西" | 🔴 P1 |
| **句式转载** | 4章用几乎相同的结尾段落 | ⚠️ P2 |
| **措辞不精确** | "连续多年登上富豪榜"→"多次位居榜首" | ⚠️ P2 |

## 五、修复规范

### 5.1 修复方式

| 修复类型 | 工具 | 适用场景 |
|----------|------|---------|
| 逐行EDIT | EDIT工具（精准替换） | 单处事实错误、措辞修正、语法断裂 |
| 脚本批量 | Node.js（正则+语境感知替换） | "就是"→"正是"/"便是" 全书348处 |
| 两轮脚本 | Node.js + 手动修复残留bug | ch106-110"够了"从379次→60次 |

### 5.2 修复优先级

1. 🔴 **硬伤** — 事实错误、语法断裂、角色能力违规（如苏墨使用绯墨的"墨迹通晓"）
2. ⚠️ **重复文字超标** — "够了"/"是最"/"就是"等
3. 🟡 **跨章衔接** — 统稿发现的空间逻辑/时间锚点/伏笔断裂
4. 🟢 **低优先级叙事改进** — 句式多样性、情感弧度、段落节奏

### 5.3 绝对禁止

- ❌ 误删系动词"是"（"X极其Y的" → 必须是"X是极其Y的"）
- ❌ 删除完整段落
- ❌ 改变递进式句法风格骨架
- ❌ 修改知识内容/事实信息
- ❌ 破坏已完成的裂隙缝补内容
- ❌ "顺便"改审稿没提到的地方

## 六、番外篇流水线

### 6.1 何时需要番外？

当正篇写完后，如果：
- 考研知识点覆盖率不足
- 大纲中某段弧线被跳过（如中世纪+文艺复兴）
- 某类知识域在正篇中完全空白（如外国文学19-20世纪）

### 6.2 番外篇标准流程

```
庄方宜出大纲 → 凯尔希审核 → 赫默写正文 → 真理核查 → 符玄审计 → 赫默返修 → commit
```

### 6.3 番外篇大纲模板

每篇大纲必须包含：
1. **定位** — 补遗还是扩展？覆盖多少知识点？
2. **叙事框架与衔接设计** — 怎么融入正篇世界观？触发事件是什么？
3. **每章详细大纲** — 标题、视角、概要、知识点、NPC设计
4. **风格约束** — 与正篇相同的阈值

### 6.4 知识卡片格式

当某个知识域不需要叙事弧线时，用卡片代替：

```markdown
# Card XX: 考点名称
## 核心定义（一句话定义）
## 关键人物（人名+一句话贡献）
## 代表作品/文献
## 考研出题方向
## 关联考点
```

## 七、Session管理

### 7.1 命名规范

```
{角色}-{项目}-{任务描述}
例：赫默-纸上宇宙-ch101-105
    符玄-纸上宇宙-ch86-90审稿
    庄方宜-纸上宇宙-ch91-100统稿
```

### 7.2 生命周期

- **一次性审稿Session**：完成即丢弃
- **写作Session**：可复用（同一个赫默Session可以连续写多个批次）
- **总控Session**：凯尔希始终活跃

### 7.3 巡检机制

```bash
cron_schedule --delay "30m" --repeat "30m" --tag "纸上宇宙-巡检"
```

巡检任务会自动检查所有子Session的进度，完成后 commit + push。

## 十、经验教训

### 10.1 做得好的

1. **5章一批+双审并行** — 节奏稳定，不会堆积审稿压力
2. **批量模式修复用脚本** — "就是"348处、ch106-110"够了"379→60次，手工不可行
3. **审稿Session一次性用完即弃** — 避免session膨胀
4. **庄方宜的大纲质量** — 番外一至八的大纲精确到每章知识点，赫默写起来不会偏离

### 10.2 踩过的坑

1. **"就是"全书性超标被发现得太晚** — 应该在做完第一批正篇后就查，而不是等到120章全写完
2. **重复文字修复误删系动词** — 批量替换"是最"时把"是"一起删了，导致语法断裂。修复规则必须加上"检查系动词是否还完整"
3. **番外一句式链锁** — "X是Y的。Y是Z的。"11.3%——番外一写得太像字典。后续番外用番外五作为范本
4. **Session被回收导致产出丢失** — 多次出现赫默session跑完但0 turns并idle。产出文件落盘了但session状态异常。定期用git status检查实际文件产出
5. **元叙事泄露** — ch119/ch120角色引用了"第九十二章"这种章节号。角色不应该知道自己在第几章

### 10.3 值得保留的模式

- **裂隙缝补机制** — 统稿（庄方宜）发现跨批裂隙 → 赫默精准修复 → commit
- **白面鸮数据验证** — 用bash grep做批量统计比人工检查快100倍
- **四路最终审查** — 逻辑审计+文笔润色+可读性+考点覆盖，四个维度不冲突
- **知识卡片附录** — 100张卡片覆盖292条LEC知识点，不需要写叙事弧线
